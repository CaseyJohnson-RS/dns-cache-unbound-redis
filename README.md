# Кэширование DNS — Unbound + Redis

Реализация рекурсивного DNS-резолвера с внешним кэшем и управляемым временем хранения записей.
Кейс конкурса **«Твой проект» 2025/2026**, направление «Системный анализ и суперкомпьютерное моделирование».

**Документация:** [caseyjohnson-rs.github.io/dns-cache-unbound-redis](https://caseyjohnson-rs.github.io/dns-cache-unbound-redis/)

---

## О проекте

Стандартный DNS-резолвер хранит записи ровно столько, сколько указано в TTL авторитетного сервера. Если связь с авторитетным сервером обрывается после истечения TTL — ответы недоступны. Цель проекта — показать, как можно хранить DNS-ответы сколь угодно долго с помощью внешней базы данных, а также продемонстрировать управление DNSSEC-валидацией через локальные зоны.

## Стенд

| Компонент | Версия |
|---|---|
| ОС | Ubuntu Server 22.04 |
| Unbound | 1.24.2 (сборка из исходников, `--prefix=/usr`) |
| Redis | из пакетов apt |

Конфигурация: `/etc/unbound/unbound.conf`, бинарь: `/usr/sbin/unbound`.
Модули: `validator cachedb iterator`.

## Что реализовано

**1. Кэширование стандартными средствами**
- Наблюдение за убыванием TTL в кэше резолвера
- Prefetch — принудительное обновление записей до истечения TTL
- `cache-min-ttl` — нижняя граница времени хранения записей
- Обнаружение принудительной очистки кэша при перезапуске Unbound

**2. Внешний кэш на Redis**
- Подключение Redis через модуль `cachedb` (`backend: "redis"`)
- Проверка сохранения записей в Redis при первом запросе
- Восстановление ответов из Redis после перезапуска Unbound (in-memory кэш очищается, Redis — нет)
- Настройка времени хранения через `redis-expire-records` и TTL ключей

**3. Подмена зоны с DNSSEC**
- Выбор зоны с DNSSEC (`iana.org`) и проверка цепочки доверия
- Создание `local-zone: static` — резолвер отвечает из локальных данных, минуя авторитетные серверы
- Автоматический сбор данных зоны из кэша и генерация конфига (`local-zone-iana.conf`)
- Демонстрация ответов из локальной базы с трассировкой рекурсивного поиска
